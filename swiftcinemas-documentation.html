SwiftCinemas iOS Client System Documentation — March 2026

A native iOS client for the Cinemas movie ticket booking platform. Full user journey from authentication through movie discovery, venue selection, seat reservation, Braintree-powered payment, and purchase history management.

Auth

Login, registration, voucher activation, device binding, HMAC-SHA512 + AES token security

Booking

Movie browsing, venue discovery, real-time seat map, Braintree payment (Cards & PayPal)

Realtime

WebSocket notifications, Firebase Cloud Messaging, geofencing proximity alerts

Maps

MapKit venue annotations, user location tracking, geofence region monitoring

Tickets

Purchase history, QR code generation, PDF ticket export, calendar integration

Admin

Add and update screenings, manage movies and venues via admin panel

Architecture

Backend Services

The app communicates with four Quarkus microservices behind an NGINX / Apache reverse-proxy layer:

ServicePortRoot PathPurpose
dalogin-quarkus443/loginAuth gateway & session management. Proxies purchase/checkout requests to mbooks.
mbook-quarkus8888/mbook-1User, device, support API. WebSocket endpoint for notifications.
mbooks-quarkus8080/mbooks-1Movie catalogue, booking engine, payment processing.
simple-service-webapp8085/simple-service-webappServes movie posters, venue photos, and profile pictures.

The default server URL is https://milo.crabdance.com, configured as the global constant serverURL in GeneralRequestManager.swift.

Request Flow

Purchase and checkout requests go through /login/CheckOut and /login/GetAllPurchases — these are proxy servlets in dalogin that inject session-derived headers (uuid, token2, TIME_, cookies) before forwarding to mbooks.

Security & Authentication

Login Flow

1
User enters username and password in LoginVC.
2
Password is hashed client-side with SHA3-512 via CryptoJS.SHA3().hash(password, outputLength: 512).
3
An HMAC-SHA512 signature is computed:
  • HMAC secret = hmacSHA512(username, password_hash)
  • HMAC payload = /login/HelloWorld:user={username}&pswrd={hash}&deviceId={deviceId}:{timestamp}:{body_length}
  • Result sent as X-HMAC-HASH header; timestamp as X-MICRO-TIME header.
4
POST to /login/HelloWorld with URL-encoded body: user=...&pswrd=...&deviceId=...&ios=...
5
On success (HTTP 200, success == 1):
  • JSESSIONID and X-Token stored in UserDefaults
  • ISLOGGEDIN = 1 flag set
  • Cookies (JSESSIONID, XSRF-TOKEN) managed by HTTPCookieStorage.shared

Registration Flow

1
User enters username, email, password, confirm password, and optional voucher code in SignupVC.
2
Password is SHA3-512 hashed client-side.
3
POST to /login/register with user credentials and device ID.
4
If a voucher is provided, it is validated via /login/voucher.
5
Account activation email can be triggered via /login/activation.

HMAC-SHA512 Authentication

The login request uses a two-layer HMAC scheme. All crypto operations are performed via CryptoJS JavaScript libraries executed in JavaScriptCore (CryptoJS.swift): 

hmacSecret = HMAC-SHA512(key=password_hash, data=username)
hmacHash   = HMAC-SHA512(key=hmacSecret, data=canonical_request_string)

The canonical request string format is: {path}:{body_params}:{timestamp}:{content_length}

Available CryptoJS classes

AES MD5 SHA256 SHA3 hmacSHA512

Each loads the corresponding .js file from the app bundle and evaluates it in a shared JSContext.

AES Ciphertext Token

After login, subsequent API requests include a Ciphertext header containing an AES-128 encrypted token. Encryption uses PBKDF2 key derivation: 

Key size:        128 bits
Iteration count: 1000
Salt:            3FF2EC019C627B945225DEBAD71A01B6985FE84C95A70EB132882F88C0A59A55
IV:              F27D5C9927726BCEFE7510B1BDD3D137
Passphrase:      SecretPassphrase

The token value (X-Token) is received from the server at login and stored in UserDefaults. It is encrypted by ciphertext.swift before being sent as the Ciphertext header. An additional X-Token: client-secret header and X-Device: {deviceId} header are attached to every request by GeneralRequestManager.dataTask().

Caching Strategy

LayerTechnologyWhat It CachesTTL
RealmCachedResponse (RLMObject)API responses (movies, images) keyed by URL or query1 hour
CoreDataCachedURLResponse entityWeb resources from WebView requestsManual eviction
URLCacheiOS standard URL cacheProtocol-level HTTP cachingHTTP headers

Realm caching flow in GeneralRequestManager

1
If isCacheable == "1", query Realm for a CachedResponse matching the URL/query.
2
If found and timestamp + 3600 > now → return cached data immediately (no network call).
3
If found but expired → make network call, delete stale Realm entry.
4
If not found → make network call, save response to Realm.

AppDelegate cleanup: On launch, checkRealm() evicts expired movie-paging cache entries from Realm.

Payment Processing

Payments are handled via Braintree (Cards & PayPal) through the Drop-In UI.

1
Get client token: GET /login/CheckOut → response contains clientToken
2
Present Drop-In: BTDropInController(authorization: clientToken, request: BTDropInRequest())
3
User selects payment method → Braintree returns paymentMethodNonce
4
Submit payment: POST /login/CheckOut with URL-encoded body: 
payment_method_nonce={nonce}&seatsToBeReserved={json_array}&orderId={timestamp}
5
Server processes: dalogin forwards to mbooks → Braintree server-side transaction → pessimistic seat locking → ticket creation.
6
Response contains:
  • seatsforscreen — updated seat map
  • tickets — booked ticket details
  • ResponseText, Status, Amount, TaxAmount — payment result

Seat reservation data is a dictionary of NSDictionary values (Seats global), serialised to JSON for the POST body.

Known Issues & Technical Debt

Architecture

  • Global mutable state: Most data model arrays (TableData_, SeatsData_, BasketData_, PlacesData_, etc.) are file-level global variables, creating implicit coupling between view controllers and potential race conditions.
  • Three request managers: GeneralRequestManager, RequestManager, and RestApiManager overlap in responsibility and should be consolidated.
  • No dependency injection: View controllers directly instantiate request managers and reference global data. No protocol-based abstractions for testing.

Security

  • Hardcoded AES credentials: Salt, IV, and passphrase are embedded in ciphertext.swift. Should be loaded from secure configuration.
  • Braintree sandbox token hardcoded in BasketVC.swift.
  • CryptoJS via JavaScriptCore: Running JavaScript crypto in JSContext is functional but fragile and not performant. Native Swift crypto (CryptoKit or CommonCrypto) would be preferable.

Networking

  • Force-unwrapping: Extensive use of ! on JSON parsing, data casts, and URL construction. Network failures or malformed responses will crash the app.
  • Deprecated APIs: UIAlertView is used in several places (deprecated since iOS 9).
  • No structured error handling: Errors are shown via ad-hoc alerts; no centralised error type or user-facing error messages.

UI

  • Programmatic UI: Most UI is built programmatically with hardcoded frames, not using Auto Layout. May not adapt well to all device sizes.
  • No SwiftUI: Entire app is UIKit-based. Migration to SwiftUI would modernise the codebase.

Testing

  • No unit tests: SwiftLoginScreenTests/ exists but contains minimal coverage.
  • No UI tests.

Dependencies

  • Starscream pod is still in the Podfile but replaced by native URLSessionWebSocketTask. Can be removed.
  • AFNetworking headers are included via bridging header but functionality appears unused (replaced by URLSession).

Refactoring Roadmap

Prioritised improvements specific to the iOS client, ordered by impact and risk.

Phase 1 — Safety & Correctness

🔴 HIGH — Eliminate Force-Unwrapping on Network Data

Where: GeneralRequestManager.swift (dataTask()), LoginVC.swift, MoviesData.swift, SeatsData.swift, all data model init(add:) constructors

Problem: Pervasive use of ! on data!, json["key"] as! Type, and URL(string:)!. Any malformed server response, network timeout, or missing JSON field causes an instant crash. This is the #1 source of production crashes in the app.

Fix: Replace force-unwraps with guard let / if let chains. Use SwiftyJSON's optional accessors (.string, .int) instead of as! Type casts. Return nil or display user-friendly errors instead of crashing.

// BEFORE (crash-prone)
let name = (add["name"] as! String)
let json: JSON = try! JSON(data: data!)

// AFTER (safe)
guard let data = data else { onCompletion(nil, .noData); return }
let json = try? JSON(data: data)
let name = add["name"] as? String ?? "Unknown"

🔴 HIGH — Remove Global Mutable State

Where: MoviesData.swift (TableData_), SeatsData.swift (SeatsData_), BasketData.swift (BasketData_), PlacesData.swift (PlacesData_), PurchasesVC.swift (TableData), and 10+ more file-level var declarations

Problem: ~20 global mutable variables are shared across view controllers via file-level var. Multiple VCs can read/write the same array concurrently (e.g., during screen transitions), causing data races and stale UI. Deinit methods manually call .removeAll() which can race with background network callbacks.

Fix: Introduce a DataManager (or per-domain managers: MovieStore, BookingStore, PurchaseStore) as @MainActor classes. Pass data through segues/initialisers or use Combine publishers. Remove all file-level var model arrays.

// BEFORE
var SeatsData_: [SeatsData] = .init()  // global, any VC can mutate

// AFTER
@MainActor
class BookingStore {
    static let shared = BookingStore()
    private(set) var seats: [SeatsData] = []
    func loadSeats(for screeningDateId: Int) async { ... }
}

🔴 HIGH — Externalise Hardcoded Secrets

Where: ciphertext.swift (AES Salt, IV, Passphrase), BasketVC.swift (Braintree sandbox token), GeneralRequestManager.swift (serverURL), Websocket.swift (WebSocket URL)

Problem: AES encryption constants (Salt, IV, Passphrase), Braintree sandbox token, and server URLs are committed in source code. The AES constants must match the server — any drift causes auth failures. Sandbox token should never appear in release builds.

Fix: Move secrets to xcconfig files excluded from Git, or use Xcode build configuration + Info.plist variables. Use #if DEBUG for sandbox vs production tokens. Server URL should come from environment-specific config.

// In Project-Debug.xcconfig (gitignored)
SERVER_URL = https:$()/$()/localhost
BRAINTREE_TOKEN = sandbox_dpdzm97y...

// In code
let serverURL = Bundle.main.infoDictionary?["SERVER_URL"] as? String ?? "https://milo.crabdance.com"

Phase 2 — Architecture Improvements

🟡 MEDIUM — Consolidate Three Request Managers into One

Where: GeneralRequestManager.swift (465 lines), RequestManager.swift (166 lines), RestApiManager.swift (147 lines)

Problem: Three overlapping network clients with duplicated URL building, error handling, and header injection logic. GeneralRequestManager does everything but is 465 lines of tangled caching + networking + error UI. RequestManager is a subset. RestApiManager is a singleton variant. Developers don't know which to use.

Fix: Create a single APIClient class with a clean protocol-based design. Extract caching into a ResponseCache protocol (Realm implementation). Extract error handling into a central ErrorPresenter. Use async/await for cleaner call sites.

protocol APIClientProtocol {
    func request<T: Decodable>(_ endpoint: Endpoint) async throws -> T
    func requestData(_ endpoint: Endpoint) async throws -> Data
}

class APIClient: APIClientProtocol {
    private let session: URLSession
    private let cache: ResponseCache
    private let headerProvider: HeaderProvider  // Ciphertext + X-Token + X-Device
    ...
}

🟡 MEDIUM — Replace CryptoJS/JavaScriptCore with Native Swift Crypto

Where: CryptoJS.swift (664 lines), 20+ bundled .js files, ciphertext.swift

Problem: All cryptographic operations (SHA3, HMAC-SHA512, AES-128, MD5) run JavaScript code in a JSContext. This is slow (~5× vs native), fragile (silent JSContext errors), and inflates the app bundle by ~200KB of JS files. The shared cryptoJScontext is a global mutable JSContext.

Fix: Use CryptoKit (iOS 13+) for HMAC-SHA512 and SHA256. Use CommonCrypto for AES-128 with PBKDF2. Use a Swift SHA3 library (e.g., CryptoSwift) for SHA3-512. Remove all .js files and CryptoJS.swift.

import CryptoKit
import CommonCrypto

func hmacSHA512(data: String, key: String) -> String {
    let key = SymmetricKey(data: Data(key.utf8))
    let mac = HMAC<SHA512>.authenticationCode(for: Data(data.utf8), using: key)
    return Data(mac).base64EncodedString()
}

🟡 MEDIUM — Replace Deprecated UIAlertView with UIAlertController

Where: GeneralRequestManager.swift, LoginVC.swift, HomeVC.swift, RestApiManager.swift

Problem: UIAlertView has been deprecated since iOS 9 (2015). Multiple instances create alert views from background threads with alertView.show(), which is undefined behaviour on modern iOS.

Fix: Replace all UIAlertView with UIAlertController. Ensure alerts are presented on the main thread via the topmost view controller. The existing UIAlertController.popUp() extension in MoviesData.swift is a good pattern — extend it app-wide.

🟡 MEDIUM — Introduce Structured Error Handling

Where: All networking code, all view controllers

Problem: Errors are handled ad-hoc: some show alerts, some print to console, some are silently ignored. HTTP status codes 300, 404, 503 have special-case handling in GeneralRequestManager.dataTask(). There's no centralised error type.

Fix: Define an AppError enum (network, auth, server, parsing). Create an ErrorHandler service that maps errors to user-facing messages. Use Result<T, AppError> or async/await with typed throws.

enum AppError: Error {
    case networkFailure(underlying: Error)
    case httpError(statusCode: Int, message: String)
    case authRequired
    case activationRequired(voucherActive: Bool)
    case decodingFailed
}

Phase 3 — Modernisation

🟢 LOW — Adopt async/await + Combine

Where: All networking calls, all data loading

Problem: All network calls use nested closure callbacks with DispatchQueue.main.async dispatches. This creates callback pyramids, makes error propagation difficult, and makes it hard to reason about execution order.

Fix: Wrap URLSession data tasks in async/await (available since iOS 15). Use Combine publishers for reactive data updates. This aligns with Apple's modern concurrency model and eliminates manual dispatch.

🟢 LOW — Migrate to SwiftUI (Incremental)

Where: All view controllers

Problem: The entire UI is UIKit with programmatic frame-based layout. No Auto Layout, no size classes — screens may not render correctly on all device sizes (especially iPad, Dynamic Type).

Fix: Start with leaf screens (MovieDetailVC, TicketsVC) and wrap them in UIHostingController. Gradually migrate from the outside in. Keep Storyboard.storyboard as the navigation spine until all screens are migrated.

🟢 LOW — Add Test Coverage

Where: SwiftLoginScreenTests/

Problem: Zero meaningful test coverage. No unit tests for crypto (HMAC, AES), data model parsing, or networking. No UI tests for the booking flow.

Fix: Add unit tests for: (1) HMAC-SHA512 computation with known vectors, (2) AES encryption/decryption round-trip, (3) data model init(add:) with valid and malformed dictionaries, (4) Realm caching TTL logic. Add UI tests for the login → browse → select → checkout happy path.

🟢 LOW — Clean Up Legacy Dependencies

Where: Podfile, SwiftLoginScreen-Bridging-Header.h

Problem: Starscream pod is in the Podfile but replaced by native URLSessionWebSocketTask. AFNetworkReachabilityManager.h and AFSecurityPolicy.h are imported in the bridging header but the functionality is handled by CustomURLSessionDelegate. Dead code increases build time and binary size.

Fix: Remove Starscream from Podfile. Remove AFNetworking headers and .m files from the project. Run pod install to clean up.
